# Codex Execution Prompt - Photo App Beta Upgrade

아래 기준으로 이번 작업을 진행한다.

## 1. 작업 목적

이 프로젝트는 사용자가 사진을 업로드하면 사진 분위기와 스타일을 분석하고, 문구, 캡션, 해시태그, 텍스트 포함 이미지를 생성하는 SNS 보조 앱이다.

이번 작업의 목표는 두 가지다.

1. 베타 사용자 기준으로 결과 화면 UX를 더 명확하게 정리한다.
2. 촬영 팁 기능을 두 역할로 분리한다.

반드시 지킬 것:

- 전면 재구축 금지
- 기존 provider / repository / storage 분리 구조 유지
- 유료 OpenAI API 없이도 기본 흐름 유지
- UI 텍스트는 i18n 리소스 사용
- 컴포넌트 내 한글 하드코딩 금지
- 과도한 추상화 금지
- 베타 안정성과 실제 사용성을 우선

## 2. 촬영 팁 기능 분리 원칙

촬영 팁 관련 기능은 반드시 아래 2개로 분리한다.

### A. 업로드 결과 기반 팁

역할:

- 사용자가 업로드한 현재 사진의 개선 포인트를 알려주는 기능
- 분석 결과에 종속되는 기능

특징:

- image + analysis context 필요
- 결과 화면 내부 보조 섹션으로 배치
- 현재 사진에 대한 개선 포인트만 반환
- 구도 / 빛 / 색감 / 다시 찍는 팁 / 기기 팁 포함
- 일반 촬영 가이드 카탈로그처럼 확장하지 않음

추천 라벨:

- 이 사진 개선 팁
- 사진 개선 팁

### B. 사전 촬영 가이드

역할:

- 사진을 찍기 전에 참고하는 독립 가이드
- 업로드와 무관하게 탐색 가능한 기능

특징:

- category + deviceType만으로 동작
- 상단 탭 또는 메뉴에서 진입
- 업로드 이미지, photoId, analysisId, analysis 결과 요구 금지
- 1차는 정적 카탈로그 기반
- DB repository / storage 연결 금지

추천 라벨:

- 촬영 가이드

중요:

- 두 기능을 같은 API 응답 하나로 합치지 않는다.
- 두 기능을 같은 서비스 책임으로 묶지 않는다.
- UI에서 같은 이름으로 모호하게 노출하지 않는다.

## 3. 이번 작업 범위

### P1. 결과 화면 UX 정리

우선 구현:

- 업로드 후 다음 행동이 명확하게 보이도록 CTA 정리
- 문구 복사 / 해시태그 복사 / 이미지 생성 / 이미지 다운로드 / 다시 생성 버튼 가시성 강화
- 결과 화면 정보 과부하 완화
- 업로드 결과 기반 팁 섹션 추가
- action/speed 사진에서 감성/이별/고요 문구가 섞이지 않도록 유지

### P2. 사전 촬영 가이드 1차 구현

아래 범위만 구현한다.

카테고리:

- `portrait`
- `food`
- `landscape`
- `night`
- `automotive_action`

기기 타입:

- `galaxy_s25_ultra`
- `galaxy_s26_ultra`
- `generic_smartphone`

출력:

- 조합별 3~5개 팁 카드
- `title + body` 구조

1차 제외:

- 검색
- 즐겨찾기
- DB 저장
- 사용자별 저장
- 업로드 이미지 요구
- 분석 결과 요구
- EXIF 기반 자동 기기 감지
- 무한 카테고리 확장

## 4. 권장 구현 구조

아래 구조 방향을 따른다.

```txt
services/tips/
  photo-tip-service.ts
  providers/
    mock-photo-tip-provider.ts
  strategies/
    action-speed-photo-tip-strategy.ts
    landscape-photo-tip-strategy.ts
    lowlight-photo-tip-strategy.ts
    travel-photo-tip-strategy.ts

services/shooting-guide/
  shooting-guide-service.ts
  shooting-guide-catalog.ts
  shooting-guide-types.ts
  device-guide-presets.ts
```

업로드 결과 기반 팁:

- `services/tips/*`
- 분석 결과를 입력으로 사용
- 결과 화면에서 사용
- 현재 사진의 개선 포인트만 반환

사전 촬영 가이드:

- `services/shooting-guide/*`
- `category + deviceType`만으로 동작
- 상단 탭 또는 메뉴 화면에서 사용
- 1차는 정적 카탈로그 기반
- repository / storage 연결 금지

## 5. API 응답 구조 분리

두 기능은 API 응답 구조도 분리한다.

### 업로드 결과 기반 팁

가능한 방식:

- 분석 결과와 함께 반환
- 또는 별도 endpoint에서 제공

요구:

- image + analysis context 필요
- 현재 사진에 대한 개선 포인트만 포함
- 사전 촬영 가이드 전체 카탈로그를 함께 반환하지 않음

응답 예시:

```json
{
  "photoTips": {
    "composition": "차량이 화면 아래쪽에 몰려 있어 진행 방향 여백을 조금 더 남기면 속도감이 살아납니다.",
    "lighting": "연기와 차체가 어두운 배경에 묻히는 부분이 있어 노출을 약간 올려도 좋습니다.",
    "color": "차량 색이 강한 포인트라 배경 대비를 살짝 낮추면 주제가 더 또렷해집니다.",
    "retake": "다시 찍는다면 코너를 빠져나오는 순간을 연속 촬영으로 잡아보세요.",
    "device": "Galaxy Ultra 계열에서는 3배 줌으로 조금 떨어져 찍으면 배경 압축감이 좋아집니다."
  }
}
```

### 사전 촬영 가이드

요구:

- `category + deviceType`만으로 조회
- 업로드 이미지 요구 금지
- analysis 결과 요구 금지
- 1차는 정적 카탈로그 조회
- 응답은 3~5개 카드로 제한

응답 예시:

```json
{
  "category": "automotive_action",
  "deviceType": "galaxy_s25_ultra",
  "tips": [
    {
      "title": "진행 방향에 여백 남기기",
      "body": "차가 움직이는 방향 쪽에 여백을 남기세요. 속도감이 더 강하게 보입니다."
    },
    {
      "title": "3배 줌으로 거리 확보",
      "body": "조금 떨어져 3배 줌으로 찍으면 차체 비율이 안정되고 배경 압축감이 생깁니다."
    },
    {
      "title": "연속 촬영 사용",
      "body": "드리프트나 빠른 움직임은 한 장보다 연속 촬영 후 가장 선명한 컷을 고르는 편이 좋습니다."
    }
  ]
}
```

금지:

- 하나의 `tips` 응답으로 섞고 클라이언트에서 구분하게 만들지 않는다.
- 사전 촬영 가이드 endpoint에서 `photoId`, `analysisId`를 요구하지 않는다.
- 업로드 결과 기반 팁 endpoint에서 전체 사전 가이드 카탈로그를 반환하지 않는다.

## 6. UI 진입 구조

상단 내비게이션 최소 구조:

- 사진 분석
- 촬영 가이드
- 히스토리

### 사진 분석 화면 추천 흐름

1. 사진 업로드
2. 분석 요약
3. 추천 문구/캡션/해시태그
4. 이미지 생성 및 다운로드
5. 이 사진 개선 팁

### 촬영 가이드 화면 추천 흐름

1. 카테고리 선택
2. 기기 타입 선택
3. 3~5개 팁 카드 노출

UI 원칙:

- 모바일 우선
- 첫 사용자가 이해할 수 있게 단순해야 함
- 카드 제목은 짧고 행동 중심
- 카드 본문은 1~2문장
- 결과 화면과 독립 가이드 화면의 라벨을 다르게 표시

권장 라벨:

- 결과 화면: 이 사진 개선 팁
- 독립 화면: 촬영 가이드

## 7. 콘텐츠 사용 규칙

사전 촬영 가이드는 아래 문서를 기준으로 구현한다.

- `docs/deploy/shooting_guide_content_catalog.md`

문구 스타일 원칙:

- 행동 + 이유
- 추상 표현 금지
- 촬영 행동 중심
- 기기 스펙 나열보다 실제 촬영 상황 우선
- 너무 긴 설명 금지

업로드 결과 기반 팁도 동일하게:

- 제안형 문장
- 부정적 평가 금지
- 현재 사진 개선 포인트 중심

## 8. i18n / 구조 규칙

- UI 텍스트는 리소스 파일 사용
- 컴포넌트에 한글 하드코딩 금지
- 서비스 로직과 UI 문구를 섞지 않음
- UI는 DB provider를 몰라야 함
- 서비스는 MSSQL/Postgres 구현체 직접 참조 금지
- repository interface 유지
- storage provider 교체 가능 구조 유지
- 이미지 파일은 DB에 저장하지 않음

## 11. 완료 기준

아래 질문에 모두 “예”라고 답할 수 있어야 한다.

- 처음 보는 사용자가 사진 업로드 후 다음 행동을 이해할 수 있는가?
- 결과 화면에서 문구 복사, 해시태그 복사, 이미지 다운로드가 명확한가?
- 업로드 결과 기반 팁이 현재 사진의 개선 포인트로 이해되는가?
- 사전 촬영 가이드가 업로드 없이 독립적으로 탐색되는가?
- 두 촬영 팁 기능이 UI와 코드 구조에서 섞이지 않는가?
- MSSQL/local 경로와 Postgres/Supabase 경로가 구조적으로 유지되는가?
- 유료 OpenAI API 없이도 기본 흐름이 유지되는가?

이번 작업은 전면 개편이 아니라, 기존 구조를 지키면서 베타 완성도를 빠르게 올리는 것이 목표다.
