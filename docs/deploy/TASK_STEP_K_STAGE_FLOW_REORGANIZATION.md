# TASK_STEP_K_STAGE_FLOW_REORGANIZATION.md

## 목적
현재 기능/로직은 유지한 채,
사용자 흐름을 한 화면 확장형에서 **4단계 상태 전환형 UX**로 재구성한다.

이번 작업의 핵심은 페이지를 새로 나누는 것이 아니라,
기존 `PhotoUploadScreen` / `AnalysisResultPanel` 중심 구조를 유지하면서
노출 순서와 CTA를 단계형으로 정리하는 것이다.

---

## 현재 판단

구조 판단 결과:

- **페이지 분리**: HOLD
  - 업로드 상태, 분석 결과, 선택 문구, overlayStyle, contentSet 상태 전달 범위가 커짐
  - 라우팅 / 저장 / 복원 / 상태 전달 정책이 불필요하게 커짐

- **한 페이지 안 단계형 상태 전환**: PASS
  - 기존 API, repository, storage, 분석/생성 로직 유지 가능
  - 화면 노출 순서와 CTA만 재배치하는 방식으로 안전하게 진행 가능

따라서 STEP K는
**페이지 분리 없이 단계형 상태(step state) 도입**
으로 진행한다.

---

## 목표 구조

### STEP 1. 사진 선택
- 메인 CTA:
  - 이미지 선택
  - 사진 올리기
- 보조 액션:
  - 처음 선택으로 돌아가기
  - 이전 결과 보기
  - 촬영 가이드
- 업로드 완료 후:
  - `사진 분석하기` CTA 노출

### STEP 2. 분석 결과
- 중심 노출:
  - 바로 쓰는 영역
- 보조 노출:
  - 더 보기 영역
- 이 단계에서는:
  - 느낌 바꾸기
  - 직접 꾸미기 세부 조작
  - 고급 설정
  를 최대한 줄인다
- CTA:
  - 바로 저장/복사
  - 다음: 변경하기

### STEP 3. 변경하기
- 기존 직접 꾸미기 / 선택 조정 기능을 이 단계로 집중
- 내부 구획:
  - 사진 느낌 변경
    - 분위기
	- 스타일(프리셋 / 흑백 / 필름 등)
  - 문구 변경
    - 말투
    - 감정
	- 문구 리스트
    - 캡션 리스트
- 자세히 꾸미기:
  - 기존처럼 접힘 유지
- CTA:
  - 다음: 완성하기

### STEP 4. 완성
- 중심 CTA:
  - 결과 이미지 만들기
- 완료 후 노출:
  - 완성 이미지 저장
  - 캡션 세트 복사
  - 해시태그 복사
  - 필요 시 패키지 다운로드

## 구조 규칙

다음 규칙은 유지한다.

- Tone / Style / Theme 3축 유지
- Frame은 Tone 내부
- features / domains / content 분리 유지
- UI는 로직을 가지지 않음
- 기존 API / 분석 / 생성 계약 유지
- 기존 업로드/분석/생성 흐름은 동작 그대로 유지

---

## 구현 원칙

### 1. 기존 기능을 “이동”하지 “재설계”하지 말 것
기존 기능을 새로 만들지 말고,
각 기능이 어느 단계에서 보일지만 재배치한다.

### 2. 한 단계에 한 목적만 강하게
- 사진 선택 = 올리기
- 분석 결과 = 확인하기
- 변경하기 = 다듬기
- 완성 = 저장/복사

### 3. 사용자가 현재 위치를 잃지 않게
상단에 아래 중 하나를 둔다.
- 4단계 진행 표시
- 현재 단계 강조 탭
- 이전/다음 흐름이 보이는 스텝 헤더

### 4. 숨겨진 기능이 사라진 것으로 느껴지지 않게
단계를 바꾸며 기능을 숨길 때는
반드시 다음 CTA로 연결한다.
예:
- 다음: 변경하기
- 다음: 완성하기

## 상태 설계 가이드

예시 수준:
- currentStep: 'select' | 'analyze' | 'edit' | 'finalize'

### 단계 진입 기준 예시
- select:
  - 초기 상태
- analyze:
  - 업로드 완료 + 사용자가 분석하기 선택
- edit:
  - 분석 결과가 준비된 상태에서 변경하기 선택
- finalize:
  - 변경하기 단계에서 완성하기 선택

실제 명명은 현재 프로젝트 규칙에 맞춰 조정 가능하나,
**복잡한 다중 상태 머신으로 키우지 말 것**
이 원칙이다.

## 단계별 노출 기준

### select 단계
보여줄 것:
- 업로드 입력
- 업로드 CTA
- 보조 링크

숨길 것:
- 분석 결과 상세
- 꾸미기 상세
- 완성 CTA

### analyze 단계
보여줄 것:
- 바로 쓰는 영역
- 더 보기
- 저장/복사 기본 행동
- 변경하기 CTA

숨길 것:
- 직접 꾸미기 상세 섹션 대부분
- 완성 단계 전용 CTA

### edit 단계
보여줄 것:
- 문구 변경
- 느낌 변경
- 기본 꾸미기
- 자세히 꾸미기(접힘)
- 완성하기 CTA

숨길 것:
- 업로드 주 행동
- 분석 단계 중심 CTA 일부

### finalize 단계
보여줄 것:
- 결과 이미지 만들기
- 생성 완료 상태
- 저장 / 복사 / 다운로드 CTA

숨길 것:
- 과도한 편집 옵션
- 분석 단계 중심 안내

## PASS 조건

아래를 만족하면 PASS다.

1. 전체 흐름이 `사진 선택 -> 분석 결과 -> 변경하기 -> 완성` 4단계로 구분된다.
2. 사용자가 현재 단계와 다음 행동을 이해할 수 있다.
3. 기존 API / 분석 / 생성 로직은 그대로 유지된다.
4. 기존 기능은 사라지지 않고 적절한 단계에 재배치된다.
5. 업로드 / 분석 / 직접 꾸미기 / 결과 이미지 만들기 / 저장/복사 전체 흐름이 유지된다.
6. typecheck / build 통과

가능하면 추가 확인:
- `npm run test:photo-styles`
- `npx tsx tests/photo-tips-regression.test.ts`

---

## HOLD 조건

아래 중 하나라도 있으면 HOLD다.

- step state가 과도하게 복잡해졌다
- 기능이 단계에 숨겨지며 실제로 찾기 어려워졌다
- 기존 동작 로직이 깨졌다
- 현재 단계와 다음 행동이 불명확하다
- 한 단계에 너무 많은 기능이 다시 몰렸다

## Codex 실행 프롬프트

현재 상태:
- 페이지 분리보다 단계형 상태 전환이 더 안전하다는 구조 판단 PASS
- 기존 기능/로직은 유지하고, 화면 노출과 CTA만 단계형으로 재배치하는 것이 목표

이번 작업:
- `사진 선택 -> 분석 결과 -> 변경하기 -> 완성` 4단계 흐름을 step state 기반으로 먼저 적용해
- 기존 기능은 유지하고, 단계별로 보이는 영역과 CTA만 재배치해
- 라우팅 분리는 하지 말고 한 페이지 안에서 단계 전환으로 처리해

핵심 요구:
1. 사진 선택 단계에는 `이미지 선택 / 사진 올리기`만 메인 CTA로 둔다
2. `처음 선택으로 돌아가기 / 이전 결과 보기 / 촬영 가이드`는 보조 액션으로 낮춘다
3. 분석 결과 단계는 `바로 쓰는 영역 + 더 보기 영역` 중심으로 둔다
4. 변경하기 단계는 `문구 변경 / 느낌 변경 / 기본 꾸미기 / 자세히 꾸미기` 중심으로 둔다
5. 완성 단계는 `결과 이미지 만들기 / 저장 / 복사 / 다운로드` 중심으로 둔다

수정 범위:
- PhotoUploadScreen / AnalysisResultPanel 중심 step state 및 노출 재배치
- 필요한 최소 UI 문구 보정

절대 건드리지 말 것:
- API / repository / storage
- 분석 로직
- 생성 로직
- photo tips 로직
- Tone / Style / Theme 내부 구조
- 페이지 라우팅 분리

3. 다음 단계 1개
