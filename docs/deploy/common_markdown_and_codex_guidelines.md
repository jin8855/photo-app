# 공통 지침 MD 파일

## 문서 목적
이 파일은 앞으로 Codex에게 작업을 넘길 때 공통으로 붙일 수 있는 기준 문서다.  
프로젝트가 달라도 아래 원칙은 최대한 공통으로 유지한다.

---

## 1. 문서 작성 규칙

### 기본 원칙
1. 문서는 짧은 감상문이 아니라 **작업 기준서**여야 한다.
2. “왜 하는지 / 무엇을 할지 / 어디까지 할지 / 하지 말아야 할 것”이 분명해야 한다.
3. 애매한 표현보다 결정문 형태로 쓴다.
4. 가능하면 우선순위를 명시한다.
5. 신규 스레드로 이어질 수 있게 문맥을 보존한다.

### 권장 구성
1. 배경
2. 현재 상태
3. 핵심 문제
4. 이번 작업 범위
5. 제외 범위
6. 구현 원칙
7. 우선순위
8. 완료 기준

---

## 2. Codex용 지시문 작성 규칙

### 좋은 지시문 특징
- 작업 목적이 명확함
- 기존 구조를 얼마나 유지할지 명확함
- 반드시 지켜야 할 계약이 있음
- 이번 단계와 다음 단계를 구분함
- “하지 말아야 할 것”을 적음

### 꼭 포함할 것
- 프로젝트 목적
- 현재 문제
- 원하는 결과
- 절대 변경 금지 영역
- 허용되는 변경 범위
- 결과물 형식

### 추천 문장 스타일
- “이번 작업은 전면 재구축이 아니라 구조 유지 + 기능 확장에 집중한다.”
- “기존 contract는 유지하고, 새로운 service를 추가하는 방식으로 해결한다.”
- “UI 텍스트는 리소스 파일로 분리한다.”
- “mock 구현이더라도 추후 real provider로 대체 가능한 형태여야 한다.”

---

## 3. 작업 범위 정의 규칙

### 반드시 구분할 것
- 이번에 할 것
- 이번에 하지 않을 것
- 다음 단계 후보

### 예시
#### 이번에 할 것
- 업로드 결과 화면 정리
- 새 기능 1개 추가

#### 이번에 하지 않을 것
- 전체 리팩토링
- 정식 운영 배포 자동화 전면 개편
- AI provider 완전 교체

---

## 4. 구조 관련 공통 원칙

1. **기존 구조를 함부로 무너뜨리지 않는다.**
2. controller / UI / service / repository / storage / provider 책임을 섞지 않는다.
3. 임시 mock이 실제 제품 contract를 오염시키지 않게 한다.
4. 새 기능은 가능하면 새로운 service/adapter/provider 단위로 붙인다.
5. 교체 가능성을 남기는 인터페이스가 우선이다.

---

## 5. UI 관련 공통 원칙

1. 기능이 많을수록 첫 화면은 단순해야 한다.
2. 첫 사용자 기준으로 다음 행동이 명확해야 한다.
3. 긴 설명보다 버튼과 섹션 구조가 중요하다.
4. 모바일 기준으로 먼저 본다.
5. 시각적으로 강조할 것은 1~3개로 제한한다.

---

## 6. 베타 판단 기준

베타 전에는 아래 질문으로 판단한다.

- 처음 온 사용자가 바로 사용할 수 있는가?
- 구조가 이후 확장을 막지 않는가?
- 가장 위험한 오작동 지점이 통제되고 있는가?
- 자동 저장/히스토리/상세 설정이 오히려 UX를 해치지 않는가?
- 일부가 mock이어도 전체 서비스 신뢰를 해치지 않는가?

---

## 7. 리뷰 문서 작성 규칙

리뷰 문서는 아래 구조를 기본으로 한다.

1. Critical issues
2. UX issues
3. Architecture issues
4. Performance risks
5. Data / storage risks
6. Content or generation risks
7. Recommended next fixes in priority order
8. What is already good enough

### 리뷰 작성 원칙
- 칭찬과 문제를 함께 적는다.
- 막연히 “별로다”라고 하지 말고 왜 위험한지 설명한다.
- 실제 사용자 경험 관점으로 본다.
- 개발 구조 관점도 분리해서 본다.

---

## 8. 새 스레드용 인수인계 문서 규칙

스레드가 길어질 때는 반드시 이어가기용 MD를 만든다.

### 포함할 것
1. 현재 작업 위치
2. 이미 완료한 것
3. 아직 안 된 것
4. 보류한 것
5. 다음 권장 작업
6. 절대 잃으면 안 되는 결정사항
7. 관련 파일 / 관련 URL

### 중요
- 히스토리 나열만 하지 말고 **현재 판단 기준**을 남긴다.
- 새 스레드에서 바로 이어서 일할 수 있어야 한다.

---

## 9. Codex에게 요청할 결과물 형식

Codex 작업 결과는 가능하면 아래 형식으로 요청한다.

1. 변경 요약
2. 변경 파일 목록
3. 핵심 설계 설명
4. 남은 리스크
5. 다음 단계 제안

이 형식이면 검토와 다음 지시가 쉬워진다.

---

## 10. 금지 패턴

아래는 가능하면 피한다.

- 목적 없이 파일만 대량 수정
- mock 코드가 제품 경계 안쪽까지 침투
- UI에 한국어 하드코딩 남발
- 근거 없는 과도한 AI 표현
- “일단 다 바꾸고 보자” 식 전면 재구성
- 저장/자동화 기능을 너무 일찍 과도하게 넣는 것

---

## 11. 추천 마무리 문장

Codex 지시문 마지막에는 아래 성격의 문장을 붙이면 좋다.

- “이번 작업은 작은 범위에서 완성도를 높이는 것이 목표다.”
- “정확도 100%보다 구조적 일관성과 사용자 신뢰도를 우선한다.”
- “기존 계약은 유지하고, 새 기능은 분리된 레이어로 추가한다.”
- “과한 확장보다 beta-ready 품질 확보가 우선이다.”

---

## 12. 한 줄 원칙

**지금 필요한 것은 거대한 개편이 아니라, 구조를 지키면서 사용성을 높이는 정확한 한 걸음이다.**
