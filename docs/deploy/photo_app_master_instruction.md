# MASTER - Photo App Beta Upgrade Instruction

## 1. 프로젝트 목적

이 프로젝트는 사용자가 사진을 업로드하면 사진의 분위기와 스타일을 분석하고, 그 결과를 바탕으로 문구, 캡션, 해시태그, 텍스트 포함 이미지를 생성하는 SNS 보조 앱이다.

현재 목표는 외부 베타 사용자가 설명 없이도 사용할 수 있도록 UX를 정리하고, 사진 타입에 맞는 문구 품질과 결과 화면 신뢰도를 높이는 것이다.

핵심 방향:

- 첫 사용자가 업로드부터 결과 활용까지 고민 없이 진행할 수 있어야 한다.
- 분석 결과는 과장된 AI 판단처럼 보이지 않고, 현재 사진에 대한 실용적인 도움으로 보여야 한다.
- 문구 생성, 이미지 생성, 촬영 팁은 역할을 분리한다.
- 현재는 MSSQL/local 또는 Postgres/Supabase 경로를 유지하되, UI와 서비스는 provider 구현을 직접 알지 않아야 한다.

## 2. 촬영 팁 기능 분리 원칙

촬영 팁이라는 이름을 공유하더라도 기능은 반드시 2개로 분리한다.

두 기능을 같은 데이터 흐름이나 같은 화면 책임으로 섞지 않는다.

## 3. 기능 A: 업로드 결과 기반 팁

### 역할

업로드 결과 기반 팁은 사용자가 이미 업로드한 사진의 분석 결과에 종속되는 기능이다.

목적은 “현재 사진의 개선 포인트”를 알려주는 것이다.

### 배치

- 업로드 결과 화면 내부 섹션으로 배치한다.
- 분석 결과, 문구, 캡션, 이미지 생성 흐름과 함께 노출한다.
- 독립 메뉴나 별도 상단 탭의 주 기능처럼 취급하지 않는다.

### 입력 기준

가능하면 아래 정보를 사용한다.

- `scene_type`
- `mood_category`
- `photo_style_type`
- 이미지 밝기/대비/채도 추정값
- 피사체 위치나 구도 관련 간단한 휴리스틱
- 업로드한 원본 이미지 메타데이터
- 사용자가 선택한 기기 타입이 있다면 해당 값

### 출력 항목

업로드 결과 기반 팁은 최소 아래 항목을 포함한다.

- 구도
- 빛
- 색감
- 다시 찍는 팁
- 기기 팁

### UX 원칙

- 사용자가 현재 사진을 더 좋게 만들 수 있는 짧고 구체적인 조언을 준다.
- 결과 화면의 보조 섹션으로 제공하되, 문구/다운로드 CTA를 방해하지 않는다.
- 긴 설명보다 바로 적용 가능한 한두 문장 중심으로 구성한다.
- “이 사진은 실패했다”처럼 부정적으로 보이지 않게 말한다.
- 분석 신뢰도가 낮을 수 있으므로 단정적인 표현을 피하고 제안형으로 쓴다.

### 예시

- 구도: 차량이 화면 아래쪽에 몰려 있어 속도감은 좋지만 여백이 조금 답답합니다. 다음에는 진행 방향 쪽 여백을 더 남겨보세요.
- 빛: 연기와 차체가 어두운 배경에 묻히는 부분이 있습니다. 노출을 약간 올리면 주제가 더 또렷해집니다.
- 색감: 청록색 차량이 강한 포인트라 채도는 유지하되, 배경 대비를 살짝 낮추면 차가 더 살아납니다.
- 다시 찍는 팁: 같은 장면을 다시 찍는다면 차가 코너를 빠져나오는 순간을 연속 촬영으로 잡아보세요.
- 기기 팁: Galaxy Ultra 계열에서는 3배 줌으로 조금 떨어져 찍으면 배경 압축감과 속도감이 더 좋아집니다.

## 4. 기능 B: 사전 촬영 가이드

### 역할

사전 촬영 가이드는 업로드와 무관한 독립 기능이다.

목적은 “사진을 찍기 전에 참고하는 가이드”를 제공하는 것이다.

### 배치

- 상단 탭 또는 메뉴에서 진입 가능해야 한다.
- 업로드 결과 화면 내부에 종속시키지 않는다.
- 사용자가 사진을 업로드하지 않아도 사용할 수 있어야 한다.

### 탐색 기준

사전 촬영 가이드는 카테고리와 기기 타입 기준으로 탐색할 수 있어야 한다.

1차 구현 카테고리는 아래 5개로 제한한다.

- 인물
- 음식
- 풍경
- 야경
- 자동차/액션

1차 구현 기기 타입은 아래 3개로 제한한다.

- Galaxy S25 Ultra
- Galaxy S26 Ultra
- 일반 스마트폰

1차 구현에서 제외한다.

- 검색 기능
- 즐겨찾기
- DB 저장
- 사용자별 가이드 저장
- 카테고리 무한 확장
- 고급 EXIF 기반 자동 기기 감지

### 출력 항목

사전 촬영 가이드는 업로드 결과가 없더라도 읽을 수 있는 일반 가이드여야 한다.

1차 출력은 3~5개 팁 카드로 제한한다.

카드 구성 후보:

- 추천 구도
- 빛/노출 팁
- 색감 팁
- 촬영 모드/줌 배율 팁
- 흔한 실수

카드가 너무 많아지면 첫 화면 이해도가 떨어지므로, 1차에서는 카테고리와 기기 타입 조합별로 가장 중요한 팁만 보여준다.

### UX 원칙

- 업로드 전에 참고하는 독립 콘텐츠처럼 보여야 한다.
- “내 사진 분석 결과”처럼 오해될 표현을 쓰지 않는다.
- 카테고리와 기기 타입을 바꾸면 내용이 바뀌어야 한다.
- 모바일/태블릿에서도 빠르게 탐색 가능해야 한다.
- 베타 단계에서는 과도한 기기 스펙 설명보다 실전 팁 중심으로 구성한다.

### 예시

자동차/액션 + Galaxy S25 Ultra:

- 3배 줌으로 조금 떨어져 찍으면 차체 비율이 안정적이고 배경 압축감이 좋아집니다.
- 연기나 빠른 움직임이 있다면 연속 촬영을 켜고 가장 선명한 컷을 고르세요.
- 차가 움직이는 방향 쪽에 여백을 남기면 속도감이 더 강해집니다.
- 너무 어두운 배경에서는 노출을 조금 올리되, 하이라이트가 날아가지 않게 확인하세요.

## 5. 두 기능의 경계

반드시 지킬 것:

- 업로드 결과 기반 팁은 분석 종속형이다.
- 사전 촬영 가이드는 독립 탐색형이다.
- 업로드 결과 기반 팁은 결과 화면 내부 섹션이다.
- 사전 촬영 가이드는 상단 탭 또는 메뉴에서 접근하는 별도 기능이다.
- 업로드 결과 기반 팁은 현재 사진의 개선 포인트를 다룬다.
- 사전 촬영 가이드는 사진을 찍기 전 참고할 일반 가이드를 다룬다.

금지:

- 사전 촬영 가이드를 분석 결과처럼 표현하지 않는다.
- 업로드 결과 기반 팁을 일반 콘텐츠 탐색 화면처럼 확장하지 않는다.
- 두 기능을 같은 API 응답 하나에 억지로 합치지 않는다.
- 결과 화면에서 사전 촬영 가이드 전체를 길게 노출하지 않는다.
- 독립 가이드에서 업로드 이미지가 있어야만 동작하는 구조를 만들지 않는다.

## 6. API 응답 구조 분리

두 기능은 API 응답 구조도 분리한다.

업로드 결과 기반 팁:

- 분석 결과와 함께 반환되거나 별도 endpoint에서 제공할 수 있다.
- 어떤 방식이든 image + analysis context를 필요로 한다.
- 입력은 업로드된 이미지 또는 이미지 식별자, 분석 결과 context, 선택된 기기 타입을 포함할 수 있다.
- 응답은 현재 사진에 대한 개선 포인트만 포함한다.
- 사전 촬영 가이드 카탈로그 전체를 함께 반환하지 않는다.

업로드 결과 기반 팁 응답 예시:

```json
{
  "photoTips": {
    "composition": "차량이 화면 아래쪽에 몰려 있어 진행 방향 여백을 조금 더 남기면 속도감이 살아납니다.",
    "lighting": "연기와 차체가 어두운 배경에 묻히는 부분이 있어 노출을 약간 올려도 좋습니다.",
    "color": "차량 색이 강한 포인트라 배경 대비를 살짝 낮추면 주제가 더 또렷해집니다.",
    "retake": "다시 찍는다면 코너를 빠져나오는 순간을 연속 촬영으로 잡아보세요.",
    "device": "Galaxy Ultra 계열에서는 3배 줌으로 조금 떨어져 찍으면 배경 압축감이 좋아집니다."
  }
}
```

사전 촬영 가이드:

- `category + deviceType`만으로 조회되는 별도 데이터다.
- 업로드 이미지나 analysis 결과를 절대 요구하지 않는다.
- 입력은 카테고리와 기기 타입만 사용한다.
- 1차 구현은 정적 카탈로그 조회로 충분하다.
- 응답은 3~5개 팁 카드로 제한한다.
- 사용자별 저장, 히스토리 저장, DB 저장을 요구하지 않는다.

사전 촬영 가이드 응답 예시:

```json
{
  "category": "automotive_action",
  "deviceType": "galaxy_s25_ultra",
  "tips": [
    {
      "title": "진행 방향에 여백 남기기",
      "body": "차가 움직이는 방향 쪽에 여백을 두면 속도감이 더 강하게 보입니다."
    },
    {
      "title": "3배 줌으로 거리 확보",
      "body": "조금 떨어져 3배 줌으로 찍으면 차체 비율이 안정되고 배경 압축감이 생깁니다."
    },
    {
      "title": "연속 촬영 사용",
      "body": "드리프트나 빠른 움직임은 한 장보다 연속 촬영 후 가장 선명한 컷을 고르는 편이 좋습니다."
    }
  ]
}
```

금지:

- 사전 촬영 가이드 endpoint에서 업로드 이미지, photoId, analysisId를 요구하지 않는다.
- 업로드 결과 기반 팁 endpoint에서 전체 사전 가이드 카탈로그를 반환하지 않는다.
- 두 기능을 하나의 `tips` 응답으로 합쳐 클라이언트에서 구분하게 만들지 않는다.

## 7. 권장 구현 구조

```txt
services/tips/
  photo-tip-service.ts
  providers/
    mock-photo-tip-provider.ts
  strategies/
    action-speed-photo-tip-strategy.ts
    landscape-photo-tip-strategy.ts
    lowlight-photo-tip-strategy.ts
    travel-photo-tip-strategy.ts

services/shooting-guide/
  shooting-guide-service.ts
  shooting-guide-catalog.ts
  shooting-guide-types.ts
  device-guide-presets.ts
```

업로드 결과 기반 팁:

- `services/tips/*`
- 분석 결과를 입력으로 받는다.
- 결과 화면에서 사용한다.

사전 촬영 가이드:

- `services/shooting-guide/*`
- 업로드 이미지 없이 카테고리와 기기 타입만으로 동작한다.
- 상단 탭 또는 메뉴 화면에서 사용한다.
- 1차에서는 정적 카탈로그 데이터 기반으로 동작한다.
- DB repository나 storage adapter와 연결하지 않는다.

## 8. UI 구조 원칙

### 상단 내비게이션

최소 구조:

- 사진 분석
- 촬영 가이드
- 히스토리

### 사진 분석 화면

추천 흐름:

1. 사진 업로드
2. 분석 요약
3. 추천 문구/캡션/해시태그
4. 이미지 생성 및 다운로드
5. 업로드 결과 기반 팁

### 촬영 가이드 화면

추천 흐름:

1. 카테고리 선택
2. 기기 타입 선택
3. 촬영 전 체크리스트
4. 구도/빛/색감/모드 팁
5. 흔한 실수

## 9. i18n 규칙

- UI 텍스트는 리소스 파일을 사용한다.
- 컴포넌트에 한글 문구를 직접 하드코딩하지 않는다.
- 팁 본문도 가능하면 리소스 또는 카탈로그 데이터로 분리한다.
- 서비스 로직 안에 UI 문구가 섞이지 않게 한다.

## 10. Provider / Storage / Repository 규칙

- UI는 DB provider를 몰라야 한다.
- 서비스는 MSSQL/Postgres 구현체를 직접 참조하지 않는다.
- repository interface를 통해 접근한다.
- storage provider는 local-public과 supabase-storage를 교체 가능하게 유지한다.
- DB에는 파일 경로와 메타데이터만 저장한다.
- 이미지 파일 자체를 DB에 저장하지 않는다.

## 11. 우선순위

### P1

- 결과 화면 CTA 정리
- 업로드 결과 기반 팁 섹션 추가
- 분석 결과와 문구/캡션 품질 신뢰도 개선
- action/speed 사진에 감성/이별/고요 문구가 섞이지 않게 유지

### P2

- 사전 촬영 가이드 독립 화면 추가
- 카테고리/기기 타입 기반 탐색
- 1차 범위는 인물/음식/풍경/야경/자동차·액션과 Galaxy S25 Ultra/S26 Ultra/일반 스마트폰으로 제한
- 검색/즐겨찾기/DB 저장 없이 3~5개 팁 카드만 제공
- 태블릿/데스크톱 반응형 검증

### P3

- 실제 AI vision provider 연결
- 고급 EXIF 기반 기기 감지
- 사용자별 선호 저장
- 촬영 가이드 콘텐츠 확장

## 12. 완료 기준

베타 기준 완료 판단:

- 처음 보는 사용자가 사진 업로드 후 다음 행동을 이해할 수 있다.
- 결과 화면에서 문구 복사, 이미지 다운로드, 해시태그 복사가 명확하다.
- 업로드 결과 기반 팁이 현재 사진의 개선 포인트로 이해된다.
- 사전 촬영 가이드가 업로드 없이 독립적으로 탐색된다.
- 두 촬영 팁 기능의 역할이 UI와 코드 구조에서 섞이지 않는다.
- 현재 MSSQL/local 경로와 Postgres/Supabase 경로가 구조적으로 유지된다.
- 유료 OpenAI API 없이도 기본 흐름이 깨지지 않는다.
