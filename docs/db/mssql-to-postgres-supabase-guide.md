# MSSQL to Postgres (Supabase) conversion guide

## Scope

Current runtime schema sources:

- MSSQL bootstrap: [db/mssql/init.ts](/C:/EtcProject/photo-caption-local-app/db/mssql/init.ts)
- SQLite reference schema: [db/schema.sql](/C:/EtcProject/photo-caption-local-app/db/schema.sql)
- Postgres draft: [db/postgres/schema.sql](/C:/EtcProject/photo-caption-local-app/db/postgres/schema.sql)

This guide focuses on the two application tables currently used by the app:

- `photos`
- `analyses`

## Recommended migration strategy

1. Keep the repository interfaces unchanged.
2. Add a Postgres adapter behind the existing repository bundle factory.
3. Move MSSQL-specific bootstrapping and query execution behind the adapter layer only.
4. Migrate JSON string columns to native `jsonb`.
5. Standardize all timestamps to UTC `timestamptz`.

## Type conversion guide

| MSSQL | Current usage | Postgres / Supabase | Notes |
|---|---|---|---|
| `INT IDENTITY(1,1)` | `photos.id`, `analyses.id` | `bigint generated by default as identity` | Safe default for future growth. `integer generated ...` is also possible if ids stay small. |
| `INT` | `photo_id`, score fields | `bigint` for FK, `integer` for scores | FK type must match referenced PK type. |
| `NVARCHAR(255)` | `photos.original_name` | `text` | Postgres `text` is usually simpler than length-capped varchar. |
| `NVARCHAR(500)` | `photos.file_path`, `generation_warning` | `text` | Path strings are better left unbounded unless strict validation exists. |
| `NVARCHAR(100)` | `scene_type`, `mood_category`, `recommended_text_position` | `text` | Can later be constrained with check/enum if needed. |
| `NVARCHAR(MAX)` | `long_review`, JSON columns | `text` or `jsonb` | For review text use `text`; for structured payloads use `jsonb`. |
| `DATETIME2` | `created_at` | `timestamptz` | Prefer timezone-aware timestamps for Supabase. |
| `SYSUTCDATETIME()` | timestamp default | `timezone('utc', now())` | Explicit UTC default. |
| `FOR JSON PATH` output | repository result serialization | not needed | Postgres repositories should map rows in application code rather than SQL JSON projection. |

## Required conversion details

### 1. `datetime`

Current MSSQL:

- `created_at DATETIME2 NOT NULL DEFAULT SYSUTCDATETIME()`

Recommended Postgres:

- `created_at timestamptz not null default timezone('utc', now())`

Why:

- Supabase and Postgres work best with timezone-aware timestamps.
- This removes ambiguity when later adding auth, usage limits, and billing windows.

### 2. `nvarchar`

Current MSSQL:

- Most string columns use `NVARCHAR(...)`
- Large bodies use `NVARCHAR(MAX)`

Recommended Postgres:

- Use `text` for nearly all user/content strings.

Why:

- Postgres `text` is fully Unicode-safe.
- It avoids unnecessary length bookkeeping during migration.
- Validation can stay at the application level.

### 3. `identity`

Current MSSQL:

- `id INT IDENTITY(1,1) PRIMARY KEY`

Recommended Postgres:

- `id bigint generated by default as identity primary key`

Why:

- Modern standard syntax.
- Cleaner than legacy `serial` / `bigserial`.
- Better long-term compatibility with Supabase defaults.

### 4. `json`

Current MSSQL:

- `phrases_json NVARCHAR(MAX)`
- `captions_json NVARCHAR(MAX)`
- `hashtags_json NVARCHAR(MAX)`

Recommended Postgres:

- `phrases_json jsonb not null default '[]'::jsonb`
- `captions_json jsonb not null default '[]'::jsonb`
- `hashtags_json jsonb not null default '[]'::jsonb`

Why:

- These values are structured application payloads, not plain text.
- `jsonb` supports validation, operators, and GIN indexes.
- Future filtering like "contains hashtag" or "top-scored phrase exists" becomes easier.

## Supabase-specific notes

### Good defaults

- Use schema `public` first unless there is a strong reason to separate application schemas.
- Use `jsonb` for AI output payload columns.
- Use `timestamptz` everywhere.
- Add GIN indexes only on JSON columns that will actually be queried.

### Recommended next step for app code

Current MSSQL repositories rely on SQL Server JSON projection and `sqlcmd`.

For Postgres/Supabase:

1. Add a Postgres repository adapter.
2. Map rows in TypeScript instead of emitting JSON from SQL.
3. Keep the existing repository interfaces:
   - `photoRepository`
   - `analysisRepository`

## Migration risks to watch

### JSON payload shape drift

The app has evolved from string arrays to scored phrase/caption payloads. Existing old rows may not deserialize cleanly if migrated blindly.

Recommendation:

- Normalize old rows before migration, or keep compatibility mappers in place during transition.

### Timestamp semantics

MSSQL currently stores UTC using `SYSUTCDATETIME()`. Preserve that assumption in Postgres and avoid `timestamp without time zone`.

### Identity and FK alignment

If `photos.id` becomes `bigint`, `analyses.photo_id` must also become `bigint`.

## Proposed next implementation step

1. Add `postgres-local` or `supabase` as a runtime database provider.

   - `repositories/postgres/postgres-analysis-repository.ts`
3. Reuse existing repository interfaces without changing services.
